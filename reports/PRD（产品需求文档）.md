## 1. Executive Summary

**中试 AI 助手**是一个基于大语言模型的智能生产管理聊天平台，让业务员和技术工程师通过自然语言自助查询订单进度、产能负荷、下单推荐和交期估算，替代当前"电话/微信咨询计划岗"的纯人工模式。

核心价值主张：**输入订单号 → 秒级获取全工序进度，无需等待人工回复。**

当前已完成核心技术栈搭建（pi-coding-agent + Hono SSE + Next.js + SQL Server 直连），即将进入内部试运行阶段。本 PRD 定义试运行范围、成功指标和正式上线的前置条件。

## 2. Background & Context

### 2.1 当前痛点

| 痛点 | 现状 | 影响 |
|------|------|------|
| **计划岗被打断频繁** | 业务员每次查单都需电话/微信联系计划岗，计划岗每天处理 20-50 次重复查询 | 计划岗效率低，核心排产工作被打断 |
| **信息获取延迟** | 人工查询从提问到回复平均 5-30 分钟 | 业务员无法及时回复客户交期询问 |
| **信息孤岛** | 订单进度分散在 4 个业务系统的 SQL 表中（广州中试、小试、上海中试、OA辅助单），人工跨系统查询效率低 | 同一订单在不同基地的状态需要分别询问 |
| **产能决策靠经验** | 业务员选择下单基地时缺乏数据支撑，凭经验或关系选择 | 部分基地超负荷、部分基地闲置，整体产能利用率不均 |
| **无历史追溯** | 口头/微信沟通无记录，后续扯皮时无法回溯 | 信息责任不清 |

### 2.2 项目起源

- **2026-06-26**: 从 Python FastAPI 版重写为 TypeScript Monorepo，引入 pi-coding-agent 框架解决多模型绑定、手写 tool-call 循环、提示词硬编码、会话无持久化四类架构问题
- **2026-07 初**: 核心功能可用（订单查询直连 SQL Server、产能分析、下单推荐、交期估算），进入试运行准备

### 2.3 技术架构（当前）

```
用户浏览器 (Next.js 前端)
    │  SSE (text_delta / tool_start / tool_end / thinking_delta)
    ▼
Hono Server (:8000)
    │  createSession() → AgentSession
    ▼
pi-coding-agent (工具调度 + Skills + 会话持久化)
    │  defineTool (6 个领域工具)
    ├──▶ SQL Server (mssql) — 订单进度真实数据
    └──▶ LLM API (DeepSeek) — 自然语言理解 + 回答生成
```

## 3. Objectives & Success Metrics

### 3.1 Goals

| # | Goal | Measurement |
|---|------|-------------|
| G1 | 业务员可自助查询订单进度，无需联系计划岗 | 计划岗收到的查单咨询量下降 ≥ 50% |
| G2 | AI 助手成为日常工具 | 日均活跃用户数达到目标基数的 60%+ |
| G3 | 查询响应满足业务节奏 | 端到端延迟（输入→首字响应）P95 < 5 秒 |
| G4 | 信息准确可信 | 工具返回的 SQL 数据准确率 100%（不经过 LLM 篡改）；LLM 总结的文本中无幻觉捏造数据 |

### 3.2 Non-Goals (明确不做)

| # | Non-Goal | Reason |
|---|----------|--------|
| N1 | **不修改生产数据** | 系统定位为只读查询助手，不做排产、不改订单状态、不下单。所有写操作被 `permissionGuard.ts` 硬拦截 |
| N2 | **不替代 ERP/生产管理系统** | 本系统是查询入口，不是 ERP。数据源头仍在 SQL Server，本系统只读不写 |
| N3 | **不做手机 App** | 试运行阶段仅 Web 端。移动端通过浏览器访问（响应式适配），不开发独立 App |
| N4 | **不接入 CRM 完整鉴权** | 试运行阶段使用轻量鉴权（内部白名单），正式上线前完成 CRM SSO 对接 |
| N5 | **不支持多语言** | 仅中文。用户群体为中试车间内部人员 |

## 4. Target Users & Segments

### 4.1 用户画像

| 角色 | 人数（估算） | 核心场景 | 使用频率 | 权限级别 |
|------|-------------|---------|---------|---------|
| **营销中心业务员** | ~10 人 | 查自己订单进度、问交期、选基地下单 | 高频（每天 3-10 次） | `sales` — 可查全基地订单，不可查他人客户敏感信息 |
| **技术中心工程师** | ~5 人 | 了解生产进度、排查异常工序、评估产能 | 中频（每天 1-3 次） | `engineer` — 可查全基地订单 + 产能数据 |
| **计划岗/中试主管** | ~3 人 | 全局生产状态、产能负荷、异常订单预警 | 高频（每天 5-15 次） | `planner` — 本基地全权限 / `supervisor` — 全基地全权限 |

### 4.2 用户当前 Workaround

- **业务员**: 微信/电话问计划岗 → 等回复 → 转告客户。紧急时反复催促。
- **工程师**: 亲自到车间看白板/问现场班长。
- **管理层**: 周会上听汇报，依赖计划岗整理的数据。

## 6. Solution Overview

### 6.1 核心设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 交互模式 | 聊天对话（Chat），非表单/命令行 | 自然语言门槛最低，适合非技术用户；LLM 可处理歧义和追问 |
| 数据获取 | LLM 调度 Tool → SQL Server 直连 | 避免 LLM 幻觉：进度数据强制通过工具从数据库读取，LLM 仅做自然语言理解和格式化 |
| 前端框架 | Next.js (App Router) | 与团队已有技术栈一致；SSR 可用于未来的看板页面 |
| 后端框架 | Hono + SSE | 轻量；SSE 原生支持流式文本输出；与 pi-coding-agent 框架兼容 |
| LLM 供应商 | DeepSeek（默认）/ 可切换 | 中文理解能力强；成本可控（约为 GPT-4 的 1/10）；支持多供应商切换 |
| 会话持久化 | JSONL 文件（~/.pi/agent/sessions/） | pi-coding-agent 框架内置；支持分支/fork；人类可读；无需额外数据库 |
| 权限模型 | 角色矩阵（sales/engineer/planner/supervisor）+ 基地归属 | 最小权限原则；计划岗仅看本基地；禁止写操作硬编码 |

### 6.2 信息架构

```
🏠 首页 (/)
├── 💬 AI 对话 (/chat)          ← 核心入口
│   ├── 订单进度查询
│   ├── 产能负荷分析
│   ├── 下单基地推荐
│   └── 交期估算
├── 📊 生产看板 (/dashboard)     ← 全局总览
│   ├── KPI 卡片（总览）
│   ├── 基地工序分布
│   └── 逾期/预警订单
├── ⚙️ 基地管理 (/bases)         ← 后台配置（主管）
└── 📈 产能分析 (/capacity)      ← 独立分析页
```

### 6.3 工具与数据源映射

| 工具 | 数据源 | 数据时效 | Fallback |
|------|--------|---------|----------|
| `query_order_progress` | SQL Server 直连（4 套模板） | 实时 | Mock 数据（标注"模拟"） |
| `search_orders` | Mock（待接入真实搜索） | N/A | — |
| `get_production_stages` | 硬编码常量 | 静态 | — |
| `analyze_capacity` | SQL Server 待处理数 + Mock 设备参数 | SQL 部分实时 | 全量 Mock |
| `recommend_base` | Mock 设备参数 | 静态 | — |
| `estimate_delivery` | Mock 设备参数 | 静态 | — |

### 6.4 关键技术约束

- **Agent 工具白名单**: LLM 仅可调用 6 个领域工具，不能访问文件系统/Shell/网络
- **工具调用强制**: 订单进度必须通过 `query_order_progress` 获取，禁止 LLM 凭记忆回答
- **错误降级**: SQL 连接失败不崩溃，回退到 mock 数据 + 明确提示用户
- **防注入**: 所有 SQL 使用参数化查询（`@orderNo` / `@since`）
