# 中试 AI Agent 方案论证

> 中试车间智能问答与下单推荐系统 — 方案论证与立项申请

---

## 一、项目概述

本项目旨在建设**中试车间 AI Agent 智能助手**，覆盖三大核心场景：

### 1.1 用户范围

系统服务于三大部门：

| 部门 | 主要角色 | 核心诉求 |
|------|----------|----------|
| **中试** | 计划岗、主管 | 掌握全局生产状态，减少查单/回复工作量 |
| **营销中心** | 业务员 | 查询订单进度、获取下单推荐 |
| **技术中心** | 工程师、技术主管 | 了解生产进度、产能情况，支持技术决策 |

### 1.2 核心功能

| # | 核心功能 | 解决的问题 | 预期效果 |
|---|---------|-----------|---------|
| **1** | **订单进度自助查询** | CRM 只显示"生产中"，业务人员无法自助获取，频繁咨询计划岗 | 输入订单号即可查询当前工序、负责人、计划日期，**解决 60% 日常咨询** |
| **2** | **智能下单推荐** | 下单无产能约束，凭经验选基地，导致忙闲不均、交期不可控 | AI 综合负荷、产品能力、距离、紧急度等多因素推理，**推荐最优基地并解释原因** |
| **3** | **生产执行看板** | myexcel 数据分散，**连计划岗自己都看不清全局生产状态** | 实时汇总各基地订单进度、工序状态、负责人，**一屏看全局** |

**一句话定义：** 通过 AI Agent 实现**"查进度、选基地、看全局"**——订单进度一问即答，下单推荐智能决策，生产执行一屏总览，实现从"信息不透明 + 随意下单"到"数据驱动的智能管理"。

## 二、业务痛点分析

### 2.1 现状

**现状一：下单无产能约束**

当前中试订单下单流程**没有产能约束机制**，用户自行选择下单基地，不受任何系统引导。五大基地月均处理约 **25,000 单**（每基地约 5,000 单/月），但分配完全依赖个人经验和习惯。

**现状二：生产信息不透明**

现有 CRM 移动端仅笼统显示订单"**生产中**"，无法展示：
- 当前生产环节（在哪个工序）
- 对应负责人
- 计划生产日期

详细的生产进度信息存储在公司内部的 **myexcel 系统**中，但 myexcel 数据分散、缺乏汇总视图，**连中试计划岗自己都无法快速掌握全局生产情况**——需要手动翻查、拼凑信息才能回答业务人员的咨询。

业务人员无法直接访问 myexcel，只能**频繁咨询中试计划岗**，而计划岗也需要花时间查询整理后才能回复，形成**双重低效**。

**计划岗日常工作量（以广州基地为例）：**
- 广州基地平均每天 **200 次**信息咨询（查进度、问负荷、确认交期等）
- 每次人工分配/查单平均耗时 **2 分钟**
- 单基地：**200 次/天 × 2 分钟 = 400 分钟 ≈ 6.7 工时/天**
- 年化：约 **1,675 工时/年**（按 250 个工作日计）

> 6.7 工时/天 ≈ 0.84 个全职人力，仅广州一个基地的计划岗工时占用就近乎一个全职岗位。

> 其中约 60% 的咨询是单纯的进度查询（"我的订单到哪步了？"），完全可以由 AI 自助解答。

**现状三：数据基础薄弱**

五大基地虽然都有排产数据，但关键决策数据**严重缺失**：

| 数据项 | 广州 | 上海 | 天津 | 武汉 | 成都 | 说明 |
|--------|:----:|:----:|:----:|:----:|:----:|------|
| 排产数据 | ✅ | ✅ | ✅ | ✅ | ✅ | 各基地都有 |
| **生产工序数据** | ✅ | ✅ | ❌ | ❌ | ❌ | **仅广州、上海有完整工序** |
| **产品能力矩阵** | ❌ | ❌ | ❌ | ❌ | ❌ | **全部基地都没有** |
| **产能数据** | ❌ | ❌ | ❌ | ❌ | ❌ | **全部基地都没有** |



**现状四：过度依赖计划师个人经验**

下单分配决策高度依赖资深计划师的"脑中经验"——哪个基地擅长什么产品、哪条产线最近有空档、哪个客户之前延期过……这些关键知识**没有沉淀在系统中，而是沉淀在人脑中**。

这带来三个风险：

| 风险 | 说明 |
|------|------|
| **人员依赖** | 计划师请假/离职，决策质量断崖式下降；新人培养周期长，无法快速补位 |
| **经验不可复制** | 老师傅的"手感"无法传授给系统，知识留存在个人而非组织 |
| **老人也要查数据** | 即便是资深计划师，每次决策仍需花大量时间查 myexcel、打电话确认，**经验只能缩小排查范围，不能替代数据验证** |

> **本质矛盾**：经验让计划师"知道该查哪里"，但仍需要"实际去查"才能确认。AI Agent 可以同时具备"经验"（知识沉淀）和"即时查"（实时数据）的能力——既不会请假，也不需要打电话。

### 2.2 核心问题

| 问题 | 表现 | 影响 |
|------|------|------|
| **生产信息不透明** | CRM 只显示"生产中"，myexcel 数据分散，**连计划岗自己都看不全** | 业务人员频繁咨询，计划岗疲于查单回复，双重低效 |
| **无分配标准** | 下单基地选择无规则约束，全凭个人偏好 | 分配结果不可预测，管理无法把控 |
| **时效慢** | 热门基地排队长，冷门基地闲置 | 整体交期拉长，客户满意度下降 |
| **运费多** | 未考虑基地与客户地理位置关系 | 物流成本非必要增加 |
| **忙闲不均** | 部分基地超负荷运转，部分基地产能浪费 | 资源利用率低，人效不均衡 |
| **交期不可控** | 无实时负荷数据，无法预估交期 | 承诺交期频繁失信 |
| **急单插不进** | 高负荷基地无法灵活调配 | 紧急订单响应慢，影响客户关系 |
| **数据基础薄弱** | 产品能力矩阵、产能数据**全部缺失**；工序数据仅广州/上海完整 | AI 智能推荐无法全面落地，需分阶段实施 |
| **过度依赖计划师** | 决策经验沉淀在人脑而非系统，新人无法快速上手，老人也要花时间查数据验证 | 人员变动风险高，知识无法组织化沉淀，决策效率受限于个人能力上限 |

### 2.3 痛点根因

**根因一：信息孤岛**
```
生产信息分散在 myexcel → 缺乏汇总视图 → 计划岗自己都看不清 → 业务人员更无法获取 → 频繁咨询 → 计划岗疲于查单回复 → 双重低效
```

**根因二：决策无系统支撑**
```
用户下单 → 凭经验选基地 → 不知道各基地实际负荷 → 分配不合理 → 交期/成本/效率全面受损
```

**综合根因：下单决策是多因素综合判断问题，生产进度是跨系统数据查询问题，但目前都没有系统支撑，全靠人的经验和手工操作。更深层的问题是：这些经验没有沉淀为组织能力，而是绑定在个人身上——人走了，能力就没了。**

## 三、为什么这个场景需要 Agent

在讨论技术方案之前，先回答一个根本问题：**这个需求是否真的需要 AI Agent？还是用传统工作流（Workflow）就能解决？**

**一句话总结：**
AI Agent 不是“更高级的模型调用”，而是用于需要持续决策、状态管理和过程推进的任务场景。当你关心的是“事情能不能被一步步做对”，而不只是“答案对不对”，才值得引入 Agent。

### 3.1 Agent 适用性五步判断法

判断一个场景是否需要 Agent，核心看是否有必要让模型**持续决策、跨步骤执行**：

| 判断步骤 | 判断标准 | 本项目的情况 |
|---------|---------|-------------|
| **① 是否"一问一答"能解决？** | 单次输入能得到完整确定结果 → 不需要 Agent | **不能。** 推荐不是简单查询，需要综合产品能力、负荷、距离、紧急度等多因素，逐步推理得出结论 |
| **② 是否需要"过程"？** | 任务天然分步骤，中途可能失败或变更 → 需要 Agent | **需要。** 推荐过程：解析需求 → 筛选有能力基地 → 查负荷 → 算交期 → 考虑物流 → 综合推荐。每一步都可能改变后续路径 |
| **③ 是否需要状态管理？** | 依赖历史信息、用户偏好、阶段结果 → 需要 Agent | **需要。** 多轮对话中用户可能补充约束（"客户在上海""这是急单"），Agent 需要累积这些信息调整推荐 |
| **④ 是否需要模型做"选择"而不是“生成”** | 选工具、选路径、决定何时停/何时问，模型需要在不确定条件下不断判断下一步→ 需要 Agent | **需要。** Agent 需判断：先查什么、哪些基地有资格、物流因素要不要纳入、结果不合理要不要重新查 |
| **⑤ 失败成本是否允许？** | 要求极高确定性、不可出错 → 不适合 Agent | 推荐 + 人工确认模式，Agent 给建议，人做最终决策，失败成本可控 |

**五步判断结论：本项目满足 Agent 的全部适用条件。**

本项目包含两大核心场景：

| 场景 | 复杂度 | 为什么需要 Agent |
|------|--------|----------------|
| **智能下单推荐** | 高 | 多因素持续推理（产品能力+负荷+距离+紧急度），不是简单查询 |
| **订单进度查询** | 低 | 虽然单步查询，但需自然语言交互、跨系统数据整合、多轮追问（"那交期呢？""能插单吗？"） |

> 订单进度查询看似可以"一问一答"解决，但实际场景中用户往往需要追问、补充条件、关联其他信息，Agent 的对话能力使其比传统查询系统更灵活。

### 3.2 具体场景论证：Agent vs Workflow 的本质差异

以一个真实下单场景为例：

> **用户："我要下一批 PC 阻燃材料的急单，客户在上海，200kg，5 天内要"**

**如果用 Workflow（固定流程）：**

```
输入: 产品=PC阻燃, 紧急=是
  ↓
Step 1: 查产品-基地矩阵 → 能做PC阻燃的基地列表 (固定规则)
  ↓
Step 2: 查这些基地的负荷 → 取最低 (固定排序)
  ↓
Step 3: 返回结果 (固定模板)
```

**问题：** Workflow 无法处理"客户在上海所以就近更好"、"如果最低负荷的基地设备在检修怎么办"、"如果两个基地都满了该怎么建议"这些**未预设的情况**。

**如果用 Agent（持续决策）：**

```
Agent 接收目标：为PC阻燃急单推荐最优基地
  ↓
Step 1: 解析约束条件
        产品=PC阻燃, 紧急=是, 客户=上海, 数量=5吨, 交期≤5天
  ↓
Step 2: 查产品能力矩阵 → 能做PC阻燃: [广州, 上海, 成都]
        ▸ Agent判断: 天津和武汉没有PC材料产线，排除
  ↓
Step 3: 查这三个基地负荷
        ▸ 广州 36.5%, 上海 22%, 成都 14.3%
        ▸ Agent判断: 三个基地负荷都正常，继续
  ↓
Step 4: 计算交期
        ▸ 广州 4天, 上海 3天, 成都 4天
        ▸ Agent判断: 都满足5天要求，交期不是区分因素
  ↓
Step 5: 考虑物流（Agent自主决定引入此因素）
        ▸ 客户上海→广州: 远, 客户上海→上海: 同城, 客户上海→成都: 远
        ▸ Agent判断: 上海基地物流优势明显
  ↓
Step 6: 综合推荐
        ▸ "推荐上海基地：具备PC阻燃生产能力，负荷率仅22%，
           预计0.2天排产，且与客户同城，物流时效最优。"
```

**关键差异：** Agent **自主决定**引入物流因素，**自主判断**设备情况以及负荷计算交期。这些决策路径不是预设的，是模型根据上下文**推理出来的**。这正是 Agent 的核心价值——**在不确定条件下持续判断下一步做什么。**

### 3.3 Agent 的扩展价值：不只是推荐

| 能力 | 说明 | 为什么 Workflow 做不到 |
|------|------|---------------------|
| **多因素动态推理** | 根据具体情况决定考虑哪些因素（有时看距离，有时看紧急度） | Workflow 只能走预设路径 |
| **异常自适应** | 用户说"上海在检修" → Agent 更新产能、重新推荐 | Workflow 未预设的分支无法处理 |
| **主动监控预警** | 检测到某基地负荷突增 → 主动建议分流 | Workflow 不能"主动做事" |
| **多轮对话积累** | 用户逐步补充约束，Agent 持续调整方案 | Workflow 无状态管理 |
| **自然语言解释** | 每次推荐理由不同，因为决策因素不同 | Workflow 只能模板化输出 |

## 四、技术方案对比：自开发 Agent vs 现成通用 Agent 产品

确认了 Agent 是合适的技术路线后，还需要回答：**是自开发业务 Agent，还是直接采购/接入 Claude、Codex、OpenClaw、WorkBuddy 这类现成的通用 AI Agent 产品？**

### 4.1 方案定义

| 方案 | 说明 | 代表产品/技术 |
|------|------|-------------|
| **方案A：自开发业务 Agent** | 围绕中试业务场景自主编码，完全掌控推理逻辑、业务工具与数据通路 | TypeScript + Pi + Node.js + 国产 LLM |
| **方案B：现成通用 Agent 产品** | 直接使用市面上的通用 AI Agent / AI 助手产品，通过接入或二次配置使用 | Claude、Codex、OpenClaw、WorkBuddy 等 |

> 方案B 的产品特点：面向通用编程/办公/对话场景，通过云端 SaaS 提供服务，自带强大的 LLM 能力和通用工具集（联网、代码、文档），但**默认不理解企业内部业务，也未必能直接读写公司内部系统业务数据。另外，权限管理需要重写**。

### 4.2 核心能力对比

| 能力维度 | 方案A：自开发业务 Agent | 方案B：现成通用 Agent 产品 |
|---------|----------------------|----------------------------|
| **产品定位** | 高度自定义的企业数字员工 | 主要面向C端，偏向个人助手 |
| **通用对话/语言能力** | 取决于所选 LLM | 强。厂商持续迭代，开箱即用 |
| **业务领域理解** | 强。Prompt、Skill、术语库均围绕中试与五大基地设计 | 弱。无法识别"PC 阻燃""急单插单""上海基地能力矩阵"等业务概念 |
| **CRM/myexcel 数据接入** | 直接对接公司内网数据库与接口，实时拉取订单、产能、工序状态 | 默认无法访问内部数据，需要确认业务系统是否可开放 MCP/CLI，将数据外发或自建桥接 |
| **业务规则与决策逻辑** | 可针对"产品-基地能力矩阵、负荷阈值、紧急度策略"做硬约束 | 仅靠 Prompt和skill 灌输，难以稳定遵循复杂业务规则 |
| **多轮上下文与长期记忆** | 短期对话窗口 + 业务长期记忆（用户偏好、历史决策、采纳数据） | 通用对话记忆，无法沉淀"张三常下 PC 单"等业务画像 |
| **嵌入 CRM 工作流** | 可深度嵌入下单页/订单详情/看板，作为业务系统组件 | 一般以独立聊天窗口形态存在，难以无缝嵌入 CRM 业务流 |
| **企业微信/IM 集成** | 自定义机器人，可对接企微、钉钉等内部 IM | 生态较好，无缝对接 |
| **确定性与可回归测试** | 业务集 + Benchmark + LLM-as-Judge，可对每次推荐进行回归 | 黑盒服务，模型与策略由厂商升级，回归不可控 |

### 4.3 工程与商业维度对比

| 工程维度 | 方案A：自开发业务 Agent | 方案B：现成通用 Agent 产品 |
|---------|----------------------|----------------------------|
| **首版上线周期** | 4-8 周（MVP） | 1-2 周（账号开通+轻量配置） |
| **开发人力** | 1-2 名全栈工程师 | 0.5 名集成 + 业务方持续调 Prompt |
| **直接成本** | LLM API 费用约 ¥0.05-0.3/次 + 自托管基础设施 | 按席位/Token 订阅，单价较高且随使用线性增长 |
| **是否被厂商绑定** | 无。LLM 与组件可替换 | 有。一旦切换需重做 Prompt、工作区、知识库 |
| **数据安全合规** | 自托管，数据不出公司内网 | 数据需上传至厂商云端；境外服务（Claude/Codex）涉及跨境合规风险 |
| **业务定制能力** | 完全可控，可任意扩展工具、规则、看板 | 受厂商功能边界限制，深度业务定制基本不可行 |
| **可观测性/审计** | 全链路日志、Hook 审计、推荐采纳率指标 | 只能拿到对话级日志，业务级审计能力弱 |
| **资产沉淀** | 代码、Prompt、Benchmark、业务知识全部留在公司 | 配置/记忆主要沉淀在厂商账户，难以迁移 |

### 4.4 主流通用 Agent 产品细分对比

| 产品 | 定位 | 优劣势 
|------|------|------
| **Claude（Anthropic）** | 通用对话 + Code/Computer Use Agent | 推理与长上下文能力强，支持工具调用，长任务表现好
| **Codex（OpenAI）** | 编程/代码任务 Agent | 代码生成与开发流执行能力强，智能体规划能力强，但长任务表现没有Claude稳定
| **OpenClaw 类（通用 AI 工作台）** | 通用 AI 助理/工作台 | 提供文档、对话、插件式工具集，开箱即用 ，但容易失忆，网关不稳定
| **WorkBuddy 类（企业 AI 助手）** | 面向办公场景的企业 AI 助手 | 集成办公套件，对话体验好，生态好
#### 各自优势

**方案A（自开发业务 Agent）优势：**
- 深度对接 CRM、myexcel 与产能数据，业务推理基于真实数据
- 数据全程在公司内网，满足安全与合规要求
- 工具链、Prompt、Benchmark 均为公司资产，可持续沉淀
- 可作为业务系统内嵌组件，深度融入下单/进度/看板工作流

**方案B（现成通用 Agent 产品）优势：**
- 通用对话与代码/办公能力强，开箱即用
- 接入周期短，业务方可自助配置 Prompt
- 模型与产品由厂商持续迭代

**两者主要差距：**
- 业务领域理解、内网数据接入、数据合规、工具可控性——这四项是**结构性差距**，不是通过 skill和Prompt 能弥补的

## 五、推荐方案设计

### 5.1 系统定位

**Agent 嵌入 CRM，AI 推荐 + 人工确认**：Agent 负责分析和建议，最终下单决策权在人。业务员无需切换系统，在 CRM 下单流程中自动获得 AI 辅助。

```
业务员在 CRM 点击"新建订单"
  ↓
Agent 自动分析五大基地 → 负荷 + 能力匹配
  ↓
CRM 弹出 AI 推荐卡片："推荐下上海基地。当前负荷率仅 22%，具备 PC 材料生产能力，预计 0.2 天可排产。"
  ↓
业务员确认 → CRM 执行下单
```

### 5.2 推荐决策逻辑（Agent 多步推理）

```
Agent 决策过程（非固定流程，模型动态判断）：

Step 1: 解析用户需求 → 提取产品、紧急度、位置、数量、交期约束
Step 2: 产品能力筛选 → 排除无能力基地（必要条件）
Step 3: 负荷分析 → 计算剩余基地的实时负荷率
Step 4: 交期估算 → 预测各基地完工天数
Step 5: 动态因素引入 → Agent自主决定是否考虑物流/设备状态/历史质量
Step 6: 综合推荐 → 生成推荐理由（每次理由可能不同）

关键：以上步骤不是固定流程，Agent 会根据具体情况
      动态决定执行哪些步骤、引入哪些因素、何时停止。
```

**Agent 核心组件说明：**

| 组件 | 职责 | 业务场景 |
|------|------|----------|
| **Session 会话管理** | 维护多轮对话状态、用户身份、当前任务上下文 | 业务员先问负荷、再问推荐、再确认交期，Agent 能记住上下文 |
| **Prompt 引擎** | 动态组装系统提示 + 任务指令 + 上下文 + 少样本示例 | 根据用户角色（业务员/计划岗）切换不同提示策略 |
| **Memory 记忆系统** | 短期记忆（对话窗口） + 长期记忆（用户偏好、历史决策） | 记住"张三常下 PC 材料单"，下次自动推荐擅长 PC 的基地 |
| **Skill 技能注册** | 模块化能力定义，每个 Skill 包含触发条件、可用工具、输出格式 | 新增"异常预警"技能时，只需注册 Skill，无需改底层代码 |
| **Guardrails 安全护栏** | 输入过滤、输出审核、权限控制、日志审计 | 防止 LLM 输出不当内容，确保推荐结果符合业务规则 |
| **LLM 推理引擎** | 意图识别 → 任务规划 → 工具调用 → 结果整合 → 响应生成 | 动态决定调用哪些工具、是否需要追问用户 |
| **Hook 层** | 工具调用前后插入逻辑，统一处理权限、日志、结果过滤 | `pre_tool_call` 检查岗位权限；`post_tool_call` 过滤敏感字段、写入审计日志 |

三大核心功能依赖以下基础数据，需要在开发前梳理和准备：

#### 当前数据现状（重要）

| 数据项 | 广州 | 上海 | 天津 | 武汉 | 成都 | 说明 |

> - 工序数据只有广州、上海完整，其他基地只有排产（无法展示详细工序进度）
> - 产品能力矩阵和产能数据**完全缺失**，需要从计划岗处人工梳理

#### 应对策略：分阶段实施

> **核心矛盾**：智能推荐必须综合五大基地数据做全局比较，不能只覆盖两个基地；但工序数据仅广州/上海完整，其他基地只有排产。因此分阶段不按基地划分，而是按**使用人群**划分。

```
Phase 1（MVP，4-6周）：广州中试团队先行试用
├── 使用人群：广州基地中试计划岗（最熟悉数据、最能验证效果）
├── 数据范围：覆盖五大基地的排产数据（所有基地都有）
├── 订单进度：广州/上海可查到工序级进度，其他基地显示排产概览
├── 智能推荐：基于排产负荷 + 客户位置，综合五大基地全局比较
└── 生产看板：五大基地排产概览 + 广州/上海工序详情

Phase 2（推广，4周）：扩展到其他基地计划岗 + 全体业务员
├── 使用人群：五大基地计划岗 + 前端业务员
├── 补齐天津、武汉、成都的工序数据（如有）
├── 补充产品能力矩阵、产能数据（计划岗人工维护）
└── 业务员端开放订单查询、下单推荐功能

Phase 3（优化，持续）：数据完善 + 算法增强
├── 所有基地工序数据完整
├── 推荐算法加入产品能力、物流距离、产能等因素
└── 看板展示五大基地完整生产详情
```

| 维度 | Phase 1（试用） | Phase 2（推广） | Phase 3（优化） |
|------|:---:|:---:|:---:|
| **谁在用** | 广州中试计划岗 | 全体计划岗 + 业务员 | 全员 |
| **看几个基地** | 五大基地都看 | 五大基地都看 | 五大基地都看 |
| **数据粒度** | 排产全局 + 工序仅广沪 | 全局工序（尽量补齐） | 全局完整工序 |
| **推荐因素** | 负荷 + 位置 | + 产品能力、产能 | + 物流时效、历史表现 |
| **目的** | 验证推荐逻辑是否合理 | 扩大使用范围、收集反馈 | 提升推荐精度 |

```powershell

# 4. 创建数据库和用户

sudo -u postgres psql -c "CREATE DATABASE agent OWNER agent;"

# 5. 验证
node --version
npm --version
psql --version

# ========== 第三步：部署 Agent 服务 ==========

# 1. 克隆项目代码
```

### 7.4 Benchmark 评测体系

> 参考《业务自建 Agent Benchmark 完全指南》方法论，建立分层、分阶段、可量化的评测体系。

**一句话理解 Benchmark：**

就像新员工入职考核，不能只看一次面试表现，而是要通过「笔试（基础规则）+ 实操（业务能力）+ 试用期考核（持续跟踪）」来综合评估。Agent 的评测也是一样——不是一次性测试，而是持续的、量化的评估过程。

---

#### 通俗比喻：AI Agent 就像新员工

假设公司新招了一个「AI 下单助手」，你要考核它能不能胜任工作：

| 考核方式 | 人类员工 | AI Agent |
|---------|---------|----------|
| **笔试** | 做选择题、填空题 | 确定性验证：检查输出格式、字段是否正确 |
| **实操考核** | 实际处理几个订单 | LLM-as-Judge：让另一个 AI 当考官，看推理过程是否合理 |
| **考官质检** | 让主管复核考官评分是否公正 | 可靠性评估：检查考官 AI 是否会「放水」或「拍马屁」 |
| **试用期上岗** | 让两位候选员工各带一批客户，比真实业绩 | AB 测试：流量切分对比新旧 Agent 的业务指标 |

#### 方法一：确定性验证 —— 检查硬性规则（笔试）

**做法**：写测试脚本，检查 Agent 输出是否符合硬性规则。

**通俗解释**：就像考试中的选择题、填空题，答案唯一，对错分明。

**业务场景举例**（PC 阻燃急单推荐）：

> 前置业务数据（用于后续三个方法统一参考）：
> - PC 阻燃能力基地：`[广州, 上海, 成都]`，天津、武汉无能力
> - 到上海客户预估运输：上海 3 天 / 广州 3 天 / 成都 4 天
> - 急单要求：交期 ≤ 4 天

```
用户输入："我要下一批PC阻燃材料的急单，客户在上海"

Agent 输出：
{
  "recommended_base": "上海基地",
  "reason": "具备PC阻燃生产能力，同城物流最快",
  "eta": "4天"
}
```

**脚本检查什么？**
- ✅ 输出是否包含 `recommended_base` 字段
- ✅ 推荐基地是否在 `[广州, 上海, 成都]` 中（天津、武汉没有 PC 能力）
- ✅ 交期是否 ≤ 4 天（急单要求）

**结果**：通过 / 不通过。不通过的直接拒绝或告警。

**局限性**：只检查格式，不检查逻辑。例如 Agent 说"推荐广州因为距离远可以多收运费"，脚本只看到基地名称对了，检查不出逻辑问题。

---

#### 方法二：LLM-as-Judge —— 让另一个 AI 当考官（实操考核）

**做法**：让一个"考官 AI"根据预设标准，对 Agent 的推理过程和输出打分（0~10 分）。

**通俗解释**：就像让经验丰富的老师傅，看新员工处理订单的过程，给个评价。

**考官关注什么？**

| 评分维度 | 说明 | 示例 |
|---------|------|------|
| 约束解析 | 是否正确理解用户需求 | 识别出"急单"、"PC阻燃"、"上海客户" |
| 推理完整性 | 是否考虑了所有因素 | 查了负荷、能力、交期、距离 |
| 多因素平衡 | 决策是否合理 | 不是只看负荷最低，还要看交期 |
| 表达清晰度 | 理由是否让人信服 | "同城物流最快"比"负荷低"更贴合急单场景 |

**高分示例**：
- Agent 推理：识别急单 → 查 PC 能力基地 → 比较负荷和交期 → 推荐上海（同城物流快，运输 0.5 天）
- 考官评分：9/10，"推理完整，急单场景下优先考虑距离"

**低分示例**（通过方法一的硬规则，但推理质量差）：
- Agent 推理：识别急单 → 只查负荷率 → 推荐广州（理由："广州 PC 产线负荷最低"）
- 输出：基地 `广州`（合法）+ eta `4天`（卡进急单约束）→ 方法一全部通过
- 考官评分：4/10，"忽略了急单+上海客户应优先同城（上海 3 天 vs 成都 4 天），多因素平衡缺失"

**注意**：考官 AI 也有偏好或不稳定的时候，所以需要方法三验证考官是否可靠。

---

#### 方法三：可靠性评估 —— 检查考官是否可靠（业务专家质检）

**做法**：故意构造"明显有问题"的 Agent 输出，看考官能否正确识别。

**通俗解释**：就像让主管复核考官的评分是否公正——给一个明显错误的案例，看考官会不会"放水"或"拍马屁"。

**测试场景**：

| 场景 | Agent 输出 | 期望考官反应 |
|------|---------|------------|
| 正常 | "推荐上海，同城物流最快" | 给高分 ✅ |
| 故意改错 | "推荐天津，本地常驻员工较多" | 给低分 ✅（考官可靠：天津无 PC 阻燃能力，硬性违规） |

**判断标准**：
- ✅ 考官给低分（如 2 分）→ 可靠，继续用
- ❌ 考官给高分（如 8 分）→ 不可靠，需要优化评估提示词

**实际建议**：正式使用前，先花半天做一次可靠性测试，确认考官不是"好好先生"。这个问题在豆包等模型上体现较多。

---

#### 方法四：AB 测试 —— 让真实业务投票（试用期考核）

**做法**：把流量按比例切给两个 Agent 版本（如旧版 vs 新版、Prompt A vs Prompt B、模型 A vs 模型 B），在真实业务中对比关键指标，用数据决定是否全量切换。

**通俗解释**：方法一/二/三都是"考场内的考试"，AB 测试是"试用期上岗"——让两个候选员工各处理一部分真实客户，看谁带来的业绩好。离线分数高 ≠ 业务效果好，最终要让真实用户投票。

**业务场景举例（沿用 PC 阻燃急单）**：

```
版本 A：当前线上 Prompt（识别急单后只比对负荷+能力）
版本 B：新 Prompt（额外引入到客户运输天数）

流量切分：50/50，灰度 2 周
观察指标：
  · 采纳率：A 72%   B 84%   (+12pp)
  · 改写率：A 21%   B  9%   (-12pp)
  · P95 时延：A 2.1s  B 2.3s  (可接受)
  · Token 成本：A ¥0.08  B ¥0.10  (+25%，可接受)

结论：B 显著优于 A，全量上线 B。
```

**AB 测试关键纪律**：

| 纪律 | 说明 |
|------|------|
| **样本量充足** | 单组业务场景建议 ≥ 200 次真实推荐，否则不下结论；急单等长尾场景需更长观察周期 |
| **分层切流** | 按业务员/产品族/基地分层切流，避免出现"A 全是简单单、B 全是复杂单"的污染 |
| **同期对比** | 必须并行跑，不可"上周 A、本周 B"——业务波动会污染结果 |
| **显著性检验** | 采纳率等比例指标用卡方/Z 检验，p < 0.05 才认为"真的有差异" |
| **回滚预案** | 设定**护栏指标**（如 P95 时延 > 3s、采纳率下跌 > 5pp）触发自动切回 100% A |
| **一次只变一个变量** | Prompt、模型、工具集不要同时改，否则归因失败 |

**与前三个方法的关系**：

```
方法一/二/三（离线，上线前）  →  方法四 AB 测试（在线，灰度期）  →  全量上线
   筛掉硬错误、推理差              业务指标真实验证                  护栏继续监控
```

离线评测是"准入考试"，AB 测试是"以业绩定岗"——两者缺一不可。

### 7.5 Agent 可观测性设计

Agent 与 Workflow 最大的不同是：**Agent 的决策过程不透明**。必须建设完整的可观测性体系，让每次推荐都**可追溯、可复盘、可审计**。

### 7.6 权限管理设计

Agent 的权限设计遵循**最小权限原则**：每个岗位只能看到与自己职责相关的数据，任何人都不能通过 Agent 修改生产数据。

#### 岗位权限规则

| 功能 | 业务员（营销） | 技术工程师（技术） | 中试计划岗 | 中试主管及以上 | 说明 |
|------|:------:|:------:|:----------:|:--------------:|------|
| **修改生产数据** | ⛔ 禁止 | ⛔ 禁止 | ⛔ 禁止 | ⛔ 禁止 | 所有人不可修改，硬拦截 |
| 查询**自己**的订单进度 | ✅ | ✅ | ✅ | ✅ | — |
| 查询**他人**的订单进度 | ❌ | ❌ | ❌ | ✅ | 主管可查全部 |
| 查询基地负荷率 | ✅ | ✅ | ✅（仅本基地） | ✅（所有基地） | 计划岗仅限本基地 |
| 查询**本基地**生产情况（工序/进度） | ✅ | ✅ | ✅ | ✅ | — |
| 查询**其他基地**生产情况 | ❌ | ❌ | ❌ | ✅ | 计划岗不可跨基地 |
| 获取下单推荐 | ✅ | ✅ | ✅ | ✅ | 营销中心 + 中试 |
| 查看生产执行看板（全局） | ❌ | ❌ | ✅ | ✅ | 主管及以上可看全局 |

> **注**：中试计划岗的「本基地」由其账号归属基地决定，登录时从 CRM Token 中读取，无需手动设置。

#### 技术实现：Hook 前置拦截

**不在业务逻辑里做权限判断**，而是在工具调用前通过 **Hook** 统一拦截，保持业务代码干净。

```
用户发送消息
    │
    ▼
Agent 解析意图 → 决定调用哪个工具
    │
    ▼
┌──────────────────────────────────────┐
│  pre_tool_call Hook（权限检查前置）   │  ← 在这里统一拦截
│                                      │
│  1. 从 Session 取出用户角色 + 归属基地 │
│  2. 判断该工具调用是否越权            │
│  3. 越权 → 返回错误，不执行工具      │
│     合规 → 放行，继续执行            │
└──────────────────────────────────────┘
    │ 放行
    ▼
工具执行（查询数据、计算负荷等）
    │
    ▼
返回结果给用户
```

**Hook 代码示例：**

```typescript
// pre_tool_call Hook：工具调用前统一权限检查
async function preToolCallHook(toolCall: ToolCall, session: Session): Promise<HookResult> {
  const { userRole, userId, baseId } = session.user  // 从 CRM Token 解析

  // ── 规则一：任何人不可修改生产数据 ──────────────────────────
  const DENY_ALWAYS = ['update_production', 'delete_', 'write_myexcel']
  if (DENY_ALWAYS.some(p => toolCall.name.includes(p))) {
    return { block: true, message: '生产数据为只读，禁止修改' }
  }

  // ── 规则二：业务员只能查自己的订单 ──────────────────────────
  if (userRole === 'sales' && toolCall.name === 'query_order_status') {
    const targetOrderOwner = toolCall.params.ownerId
    if (targetOrderOwner && targetOrderOwner !== userId) {
      return { block: true, message: '只能查询自己的订单进度' }
    }
  }

  // ── 规则三：计划岗只能查本基地数据 ──────────────────────────
  if (userRole === 'planner') {
    const targetBase = toolCall.params.baseId
    if (targetBase && targetBase !== baseId) {
      return { block: true, message: `无权查询其他基地数据，您归属 ${baseId}` }
    }
  }

  // ── 规则四：主管及以上放行所有查询 ──────────────────────────
  if (['supervisor', 'manager', 'admin'].includes(userRole)) {
    return { block: false }  // 全部放行
  }

  return { block: false }  // 默认放行（已通过前三条规则）
}
```

**Hook 注册方式：**

```typescript
// 在 Agent 初始化时注册 Hook
agent.registerHook('pre_tool_call', preToolCallHook)

// Hook 返回 block: true 时，Agent 向用户回复拒绝原因
// Hook 返回 block: false 时，继续执行工具
```

> **设计优势**：权限逻辑集中在一个 Hook 文件中，业务工具代码无需感知权限，岗位规则变更时只改 Hook，不改工具实现。
